- [CONCERN] b4e3bbeda9b7 — "phy: qcom: edp: Add eDP/DP mode switch support"

  `qcom_edp_set_voltages()` dereferences `cfg` (either
  `edp->cfg->dp_swing_pre_emph_cfg` or `edp->cfg->edp_swing_pre_emph_cfg`)
  without a NULL guard.  The old code had an explicit `if (!cfg) return 0`
  that this patch removes.  All current platform configs populate both
  fields, so there is no crash today, but any future platform that omits
  one of the two pointers will silently dereference NULL.  The defensive
  check should be restored.

  File: `drivers/phy/qualcomm/phy-qcom-edp.c`, line ~313

  Suggestion: Add `if (!cfg) return 0;` (or a WARN_ON variant) immediately
  after the cfg assignment block, before the loop over lanes.

- [CONCERN] ee41ee54b028 — "phy: qcom: edp: Add per-version LDO configuration callback"

  `qcom_edp_phy_power_on()` calls `com_ldo_config()` before
  `qcom_edp_phy_configure()` has necessarily been called, meaning
  `edp->dp_opts.link_rate` and `edp->dp_opts.lanes` may still be
  zero-initialised (from devm_kzalloc).  With link_rate=0 the condition
  `link_rate <= 2700` is true, so the eDP path silently picks the
  low-vdiff LDO value; with lanes=0 the TX1 LDO is written as 0x00.
  This is a latent issue that existed partially before (the old
  power_on path used a fixed ldo_config=0x0 for eDP), but the new
  per-version callbacks make the behaviour link_rate-dependent, so the
  risk is now higher.  The cover letter acknowledges that glymur/sc8280xp
  have not been tested; this code path is exercised on those platforms.

  File: `drivers/phy/qualcomm/phy-qcom-edp.c`, line ~1049

  Suggestion: Document in a comment that com_ldo_config() in power_on
  relies on dp_opts being pre-populated by configure(), and/or add a
  WARN_ON(edp->dp_opts.link_rate == 0) to catch the uninitialized case
  during development.

## Commit b4e3bbeda9b7: phy: qcom: edp: Add eDP/DP mode switch support

**Summary**: Splits the single `swing_pre_emph_cfg` pointer in
`qcom_edp_phy_cfg` into separate `dp_swing_pre_emph_cfg` and
`edp_swing_pre_emph_cfg` pointers, corrects several table values based on
the HPG, and updates `qcom_edp_set_voltages()` to select the right table
at runtime based on `edp->is_edp`.

### Code Logic Maps

**qcom_edp_set_voltages() — control flow**
```
1. Select cfg: is_edp → edp_swing_pre_emph_cfg, else → dp_swing_pre_emph_cfg
   (no NULL guard — see Issues)
2. Loop over lanes: compute max v_level, p_level
3. Select table row based on link_rate (<= 2700 → HBR/RBR, else → HBR2/HBR3)
4. Look up swing/emph values; return -EINVAL if 0xff sentinel hit
5. Call com_ldo_config() via ver_ops
6. Write swing/emph to tx0 and tx1 registers
7. Return 0
```

**Data flow**
- Inputs: `edp->is_edp` (set by set_mode), `dp_opts->lanes`, `dp_opts->voltage[]`,
  `dp_opts->pre[]`, `dp_opts->link_rate` (all from configure()).
- Outputs: TXn_TX_DRV_LVL and TXn_TX_EMP_POST1_LVL register writes on tx0/tx1.
- `v_level` and `p_level` are used as array indices into 4×4 tables; they are
  bounded by the loop over `dp_opts->lanes` (max 4) and the 0xff sentinel check,
  so no out-of-bounds access is possible.

**Before-vs-after delta**
- Old: single `swing_pre_emph_cfg` pointer; if NULL, return 0; if is_edp,
  override with global `edp_phy_swing_pre_emph_cfg`.
- New: two separate pointers per platform config; selected by is_edp flag;
  no NULL guard.
- Why: enables per-platform eDP and DP tables to coexist and be selected
  at runtime based on the mode set via `phy_set_mode()`.

**qcom_edp_phy_init() — delta**
- Old: `if (edp->cfg->swing_pre_emph_cfg && !edp->is_edp)` aux_cfg[8] = 0xb7
- New: `if (!edp->is_edp)` aux_cfg[8] = 0xb7
- The old condition was gated on swing_pre_emph_cfg being non-NULL (i.e. only
  platforms that had a cfg set). The new condition applies to all DP-mode
  platforms unconditionally. Since all platforms now have dp_swing_pre_emph_cfg
  set, the effective behaviour is the same, and the TODO comment is correctly
  removed.

### Issues

- **[CONCERN] Correctness — missing NULL guard on cfg pointer**
  - After removing the old `if (!cfg) return 0` guard, `cfg` is used
    unconditionally. All current platform structs populate both pointers,
    but there is no compile-time or runtime enforcement. A future platform
    that omits `dp_swing_pre_emph_cfg` or `edp_swing_pre_emph_cfg` will
    crash with a NULL dereference.
  - File: `drivers/phy/qualcomm/phy-qcom-edp.c`, line ~313
  - Suggestion: Restore a NULL check:
    `if (!cfg) return 0;` or `if (WARN_ON(!cfg)) return -EINVAL;`
    immediately after the cfg assignment.

### Minor / Style

- **[MINOR] Commit tag ordering**: `Cc: stable@vger.kernel.org` appears before
  `Fixes:` in the trailer block. Kernel convention requires `Fixes:` first.

### Positive notes

- Table value corrections (dp_pre_emp_hbr_rbr, edp_pre_emp_hbr_rbr,
  edp_pre_emp_hbr2_hbr3) are clearly motivated by the HPG and the cover
  letter cross-references the earlier discussion thread.
- Renaming `edp_phy_swing_pre_emph_cfg_v5` → `edp_phy_swing_pre_emph_cfg_v3`
  and introducing `dp_phy_swing_pre_emph_cfg_v8` for glymur is a clean
  separation of per-version tables.
- The simplification of the `aux_cfg[8]` condition (removing the TODO) is
  correct and well-motivated.

## Commit ee41ee54b028: phy: qcom: edp: Add per-version LDO configuration callback

**Summary**: Introduces a `com_ldo_config` callback in `phy_ver_ops` with
three implementations (v3/v4/v6) that program TXn_LDO_CONFIG based on PHY
version, eDP/DP mode, link rate, and lane count; replaces the previous
fixed `ldo_config = is_edp ? 0x0 : 0x1` logic in both `qcom_edp_set_voltages`
and `qcom_edp_phy_power_on`; adds `qcom_edp_phy_ops_v3` for SC7280.

### Code Logic Maps

**qcom_edp_ldo_config_v3/v4/v6() — control flow (identical structure)**
```
1. Read dp_opts->link_rate and dp_opts->lanes from edp->dp_opts
2. if !is_edp:           ldo_config = 0x0
   elif link_rate<=2700: ldo_config = <low-vdiff>  (v3:0x81, v4:0xc1, v6:0x51)
   else:                 ldo_config = <high-vdiff> (v3:0x41, v4:0x81, v6:0x91)
3. writel(ldo_config, tx0 + TXn_LDO_CONFIG)
4. writel(lanes>2 ? ldo_config : 0x00, tx1 + TXn_LDO_CONFIG)
5. return 0
```

**Data flow**
- Inputs: `edp->is_edp`, `edp->dp_opts.link_rate`, `edp->dp_opts.lanes`
- Outputs: TXn_LDO_CONFIG register writes on tx0 and tx1
- TX1 LDO is only enabled when lane count > 2 (4-lane mode), matching the
  HPG requirement noted in the v3 changelog.

**State / lifecycle**
- `com_ldo_config` is called from two sites:
  1. `qcom_edp_phy_power_on()` — before voltage/swing configuration
  2. `qcom_edp_set_voltages()` — on every configure() call with set_voltages=1
- At site 1, `dp_opts` may be zero-initialised if `configure()` has not yet
  been called (see Issues).
- At site 2, `dp_opts` is guaranteed populated because configure() copies
  opts into edp->dp_opts before calling set_voltages.

**Call graph**
```
qcom_edp_phy_power_on()
  +-- edp->cfg->ver_ops->com_ldo_config(edp)   [new]
        +-- qcom_edp_ldo_config_v3/v4/v6()

qcom_edp_phy_configure()
  +-- qcom_edp_set_voltages()
        +-- edp->cfg->ver_ops->com_ldo_config(edp)   [new]
              +-- qcom_edp_ldo_config_v3/v4/v6()
```

**Before-vs-after delta**
- Old (set_voltages): `ldo_config = edp->is_edp ? 0x0 : 0x1`; wrote same
  value to both tx0 and tx1 unconditionally.
- Old (power_on): fixed 0x0; set to 0x1 only if swing_pre_emph_cfg non-NULL
  and !is_edp; wrote same value to both tx0 and tx1.
- New: per-version callback; eDP path selects value based on link_rate;
  TX1 only written non-zero when lanes > 2.
- Why: eDP LDO programming is PHY-version-specific and link-rate-dependent
  per the HPG; the old fixed values were incorrect for eDP mode.

### Issues

- **[CONCERN] Correctness — com_ldo_config called in power_on with potentially
  uninitialised dp_opts**
  - `qcom_edp_phy_power_on()` calls `com_ldo_config()` which reads
    `edp->dp_opts.link_rate` and `edp->dp_opts.lanes`. If `configure()`
    has not been called before `power_on()`, these fields are 0 (from
    devm_kzalloc). With link_rate=0 the eDP path picks the low-vdiff
    LDO value; with lanes=0 TX1 gets 0x00. This may not match the
    hardware expectation at power-on time.
  - The old code wrote a fixed 0x0 (eDP) or 0x1 (DP) regardless of
    link_rate, so the new code changes observable behaviour at power_on
    for eDP platforms.
  - File: `drivers/phy/qualcomm/phy-qcom-edp.c`, line ~1049
  - Suggestion: Either document that configure() must precede power_on()
    and add a WARN_ON(edp->dp_opts.link_rate == 0) to enforce it, or use
    a safe default in the ldo_config callbacks when link_rate == 0.

### Minor / Style

- **[MINOR] Commit tag ordering**: `Cc: stable@vger.kernel.org` appears before
  `Fixes:` in the trailer block. Kernel convention requires `Fixes:` first.
  Suggestion: Reorder to `Fixes:` then `Cc: stable@vger.kernel.org`.

- **[MINOR] Typo in commit body**: "accroding" should be "according".
  File: commit message body, paragraph 1.

- **[NIT] `u32 ldo_config` vs driver convention**: The three ldo_config_vN
  functions use `u32 ldo_config` but the register is 8-bit and the old
  code used `u8 ldo_config`. `writel()` accepts u32 so this is not a bug,
  but it is inconsistent with the surrounding style.
  File: `drivers/phy/qualcomm/phy-qcom-edp.c`, lines ~567, ~585, ~822
  Suggestion: Use `u8 ldo_config` to match the existing driver convention.

### Positive notes

- The three-way split (v3/v4/v6) cleanly captures per-version LDO
  differences without code duplication.
- Reusing `qcom_edp_ldo_config_v6` for `qcom_edp_phy_ops_v8` (glymur) is
  correct and avoids an unnecessary fourth variant.
- The TX1 lane-count guard (`lanes > 2 ? ldo_config : 0x00`) correctly
  implements the "only enable TX1 LDO when lane count > 2" requirement.
- `qcom_edp_phy_ops_v3` reuses all v4 callbacks except `com_ldo_config`,
  which is the minimal and correct approach.
- Patch 2 has Reviewed-by and Tested-by from two independent reviewers
  (Dmitry Baryshkov, Konrad Dybcio) on real hardware (SC8280XP X13s).
