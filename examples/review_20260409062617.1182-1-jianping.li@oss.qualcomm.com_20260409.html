[BUG] Patch 3 — NULL-deref in fastrpc_buf_free() on probe-allocated remote_heap
fastrpc_rpmsg_probe() allocates data->remote_heap with kzalloc_obj(*data->remote_heap, GFP_KERNEL), leaving buf->fl = NULL, buf->dev = NULL, and buf->virt = NULL. fastrpc_buf_free() unconditionally calls dma_free_coherent(buf->dev, buf->size, buf->virt, fastrpc_ipa_to_dma_addr(buf->fl->cctx, ...)). Dereferencing buf->fl->cctx when buf->fl is NULL causes an immediate NULL-pointer dereference when fastrpc_rpmsg_remove() calls fastrpc_buf_free(cctx->remote_heap).
File: drivers/misc/fastrpc.c — probe hunk (patch 3, ~line 2378)
Do not use kzalloc_obj to create a bare fastrpc_buf stub for the probe-time remote_heap. Store the reserved-memory region as plain dma_addr_t + size_t fields directly in fastrpc_channel_ctx, or allocate via fastrpc_remote_heap_alloc() so that buf->fl, buf->dev, and buf->virt are properly initialised.
[BUG] Patch 3 — audio_init_mem flag reset without lock on err_invoke: path
Inside fastrpc_init_create_static_process(), patch 3 sets fl->cctx->audio_init_mem = true while holding cctx->lock (via spin_lock_irqsave). On the err_invoke: label the flag is reset to false without acquiring the lock. A concurrent call on another CPU that reads audio_init_mem under the lock between the invoke failure and the flag reset will see the flag as true and skip memory registration, leaving the Audio PD without its heap.
File: drivers/misc/fastrpc.c — err_invoke: label (patch 3, ~line 1377)
Reset audio_init_mem = false inside spin_lock_irqsave / spin_unlock_irqrestore to match the locking discipline used when setting it.
[BUG] Patch 3 — Memory leak when vmcount == 0 in fastrpc_rpmsg_remove()
The outer guard if (cctx->remote_heap && cctx->vmcount) means that when vmcount == 0 (no hypervisor protection), the probe-allocated remote_heap stub is never freed, leaking the kzalloc’d fastrpc_buf. The inner if (cctx->vmcount) is also redundant dead code (always true inside the outer guard).
File: drivers/misc/fastrpc.c — fastrpc_rpmsg_remove() (patch 3, ~line 2485)
Remove the redundant inner if (cctx->vmcount). Add an unconditional kfree(cctx->remote_heap) (or appropriate cleanup) for the vmcount == 0 case so the allocation is always released on channel teardown.
