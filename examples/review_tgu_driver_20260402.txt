- [BUG] 833bbebd4d03 — qcom-tgu: Add TGU decode support

  tgu_write_all_hw_regs() uses `index` as both the loop variable and the
  return-value sentinel at label `exit:`. When all loops complete successfully
  `index` holds the last valid value (>= 0) so the ternary returns 0 — fine.
  But if the very first loop is never entered (num_step == 0), `index` is
  uninitialised at `exit:`, causing undefined behaviour. The function should
  track success with a dedicated `rc` variable, not reuse the array index.

  File: drivers/hwtracing/qcom/tgu.c, line ~274

  Suggestion: Add `int rc = 0;` at function top; on error set `rc = -EINVAL;
  goto exit;`; at the label do `TGU_LOCK(); return rc;`. Remove the
  `index >= 0` ternary.

- [BUG] efb8b437a854 — qcom-tgu: Add reset node to initialize

  reset_tgu_store() drops and re-acquires drvdata->lock around the
  pm_runtime_resume_and_get() call, leaving a TOCTOU window: another thread
  can enable the TGU between the two lock sections, after which
  tgu_do_disable() silently kills a live session. Additionally,
  pm_runtime_put() is called unconditionally at the end even when the device
  was already enabled on entry (pm_runtime_resume_and_get was skipped),
  causing a PM reference-count underflow.

  File: drivers/hwtracing/qcom/tgu.c, line ~436

  Suggestion: Use a single scoped_guard(spinlock) covering the whole reset,
  consistent with enable_tgu_store(). Track whether pm_runtime_resume_and_get
  was called with a local bool so the matching put() is only issued when needed.

- [BUG] c3c8fd06f6e8 — qcom-tgu: Add signal priority support

  In tgu_probe(), sysfs_create_groups() is called before value_table is
  allocated. The sysfs show/store callbacks dereference drvdata->value_table
  directly. A userspace open between sysfs_create_groups() and the subsequent
  devm_kzalloc() for value_table will dereference NULL, causing a kernel oops.

  File: drivers/hwtracing/qcom/tgu.c, line ~602

  Suggestion: Move sysfs_create_groups() to after all value_table allocations
  are complete.

- [BUG] 3b5fa8ce6111 — qcom-tgu: Add support to configure next action

  tgu_node_visible() mutates attr->name = "default" at runtime. The attr
  pointer points into a const struct attribute_group whose attrs entries are
  compile-time-initialised compound literals. Writing to attr->name on a
  const-qualified object is undefined behaviour and will crash on architectures
  that place read-only data in write-protected pages. is_visible() may also be
  called multiple times, making the mutation non-idempotent.

  File: drivers/hwtracing/qcom/tgu.c, line ~170

  Suggestion: Introduce a STEP_SELECT_DEFAULT() macro that initialises the
  last select attribute with .attr.attr.name = "default" at compile time,
  rather than mutating the name at runtime.

- [BUG] 2c69080504f6 — qcom-tgu: Add TGU driver

  pm_runtime_enable() is never called in tgu_probe(). The AMBA bus calls
  pm_runtime_get_sync() before .probe, so the refcount starts at 1 and the
  pm_runtime_put() at probe end is correct. But without pm_runtime_enable(),
  all subsequent pm_runtime_resume_and_get() calls in enable_tgu_store() and
  reset_tgu_store() return -EACCES (runtime PM disabled), making the driver
  non-functional at runtime. pm_runtime_disable() is also missing from
  tgu_remove().

  File: drivers/hwtracing/qcom/tgu.c, line ~665 (probe) and tgu_remove()

  Suggestion: Add pm_runtime_enable(dev) before pm_runtime_put() at end of
  probe. Add pm_runtime_disable(dev) at the start of tgu_remove(), following
  the coresight ETM4x pattern.

  Calls calculate_array_location(); logs dev_err on -EINVAL; returns result.

tgu_write_all_hw_regs() [return type changed to ssize_t]:

  switch on operation_index to dispatch to priority[] or condition_decode[].

tgu_node_visible() [extended]:

Before: Only priority registers handled; tgu_write_all_hw_regs returned void.

  - File: `drivers/hwtracing/qcom/tgu.c`, line ~48

## Commit efb8b437a854: qcom-tgu: Add reset node to initialize

**Summary**: Adds a write-only `reset_tgu` sysfs attribute that zeroes all
value_table arrays and re-initialises condition_decode entries to 0x1000000
(NOT condition), then disables the TGU hardware.

### Code Logic Maps

reset_tgu_store():
  1. kstrtoul(buf); reject if != 1.
  2. spin_lock(&drvdata->lock).
  3. If !enabled: spin_unlock, pm_runtime_resume_and_get, spin_lock.
  4. tgu_do_disable(drvdata).
  5. memset priority, condition_decode, condition_select, timer, counter to 0.
  6. Loop: set condition_decode entries to 0x1000000 (NOT default).
  7. spin_unlock.
  8. pm_runtime_put(drvdata->dev).  [unconditional — BUG]
  9. return size.

Before: No way to reset TGU configuration to a known state.
After: Single sysfs write resets all configuration and disables hardware.
Why: Needed to reinitialise TGU between use cases without driver reload.

### Issues

- **[BUG] Locking & Concurrency**: TOCTOU race and PM refcount underflow.
  The lock is dropped between step 2 and step 3 to allow pm_runtime_resume_and_get().
  In the window between spin_unlock and spin_lock, another thread can call
  enable_tgu_store(val=1), which sets enabled=true and programs hardware. When
  reset_tgu_store re-acquires the lock, it calls tgu_do_disable() on a live session
  without the caller knowing. More critically, pm_runtime_put() at step 8 is
  unconditional: if the device was already enabled on entry (step 3 skipped
  pm_runtime_resume_and_get), the put() decrements the refcount below the level
  set by the caller of enable_tgu_store(), causing a PM refcount underflow and
  potentially suspending the device while it is in use.
  - File: `drivers/hwtracing/qcom/tgu.c`, line ~436
  - Suggestion: Track whether pm_runtime_resume_and_get was called with a local
    bool `need_put`. Only call pm_runtime_put() if need_put is true. Use
    scoped_guard(spinlock) for the entire operation to eliminate the TOCTOU window,
    or document clearly that reset is only safe when the TGU is not concurrently
    being enabled.
